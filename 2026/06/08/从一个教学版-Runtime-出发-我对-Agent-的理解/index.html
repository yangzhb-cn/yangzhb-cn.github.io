从一个教学版 Runtime 出发：我对 Agent 的理解

发表于2026-06-08|更新于2026-06-08

很多人第一次接触 Agent，会把它理解成“一个会调用工具的聊天机器人”。但真正把一个 Agent 项目从 TUI 做到 Web、IM、多会话、多 Agent、工具审批、上下文压缩和计划执行之后，我对 Agent 的理解发生了变化：

Agent 不是一个更聪明的 prompt，也不是一个单独的 LLM。Agent 是一套围绕 LLM 构建的运行时系统。

这个运行时系统至少包含几件事：

  • LLM：负责推理、生成、选择工具和组织语言。
  • Context：负责告诉模型“现在应该看到什么”。
  • Tool：负责把模型的意图变成真实世界的动作。
  • Session：负责保存完整、可审计的历史。
  • Event：负责让 UI、日志、通知看到运行过程。
  • Runtime：负责调度 run、队列、停止、审批、计划和后台任务。
  • UI / IM / Web：负责把 Agent 的状态变成人能理解、能控制的交互。

如果只看 LLM，Agent 像一个会说话的大脑；但如果看完整系统，Agent 更像一个“可以被观察、被控制、可恢复、可扩展的自动化执行环境”。

一、Agent 的核心不是回答，而是循环

最小的 AgentLoop 不是“用户问，模型答”这么简单，而是一条循环：

这个循环里最关键的不是调用 LLM，而是协议完整性。

模型如果返回 assistant.tool_calls，后面必须紧跟对应的 role=tool 结果，并且 tool_call_id 要一一配对。工具失败、审批拒绝、大结果卸载，也不能随便插一条普通用户消息进去，而是仍然要写回合法的 tool result。

这让我意识到：Agent 的稳定性很大程度来自“协议纪律”，不是模型能力本身。

LLM 可以犯错，但 Runtime 不能让消息协议乱掉。否则下一轮请求可能直接 400，或者更隐蔽地让模型误解上下文。

二、上下文不是 prompt，而是运行时窗口

一开始很容易把上下文理解成“把历史消息拼进去”。但会话变长后，这种做法马上会出问题：

  • 每轮请求都从 JSONL 全量 replay，性能浪费。
  • 历史越来越长，模型上下文撑爆。
  • 压缩摘要如果写进 session，会污染审计历史。
  • 动态信息如果塞进 system prompt，会和静态规则混在一起。

后来我更倾向于把上下文拆成三层：

  1. 完整历史：JSONL session，作为事实源和审计记录。
  2. 运行窗口：ContextWindowCache，保存旧对话摘要和最近完整 turn。
  3. 临时注入：<system-reminder>，把长期记忆、旧对话摘要、当前时间、Skill 等动态信息只注入本轮请求。

这三层的边界很重要。

Session 负责“发生过什么”；Context 负责“这轮应该看什么”；System reminder 负责“这轮额外提醒什么”。它们不能混在一起。

后来又加了 ContextWindowSnapshot，用于恢复上一轮的上下文压缩进度。这样重启后不用重新摘要全部 JSONL，只要验证 lastSeq/lastHash 对得上，就可以恢复 runningSummary + recentTurns，再用 loadMessageRecordsAfter(lastSeq) 补齐新增消息。

这件事让我形成了一个判断：

长会话 Agent 一定需要把“持久历史”和“请求窗口”分开。前者追求完整，后者追求可用。

三、工具不是函数，而是一套可审批的能力协议

工具调用看起来像函数调用，但在 Agent 系统里它不是简单函数。

一个工具至少要考虑：

  • 工具 schema 如何暴露给模型。
  • 参数如何解析和校验。
  • 工具是否允许并行执行。
  • 工具结果是否太大，需要卸载。
  • 工具是否高风险，需要人工审批。
  • 工具失败后如何返回合法 tool result。
  • UI 如何展示工具调用、参数、耗时和结果。

尤其是 bash/write/edit 这种高影响工具，不能让模型想调用就调用。HITL 审批不是 UI 附属功能，而是 Runtime 安全边界的一部分。

审批时要告诉用户：

  • 工具名称是什么。
  • 参数是什么。
  • call id 是什么。
  • 用户可以批准一个、拒绝一个，也可以不带 id 批量处理。

这背后还有一个协议要求：即使用户拒绝工具，也要给模型写回一个合法的 tool result，让模型知道这个工具没有执行，并继续生成回答。

所以我现在理解的工具系统，不是“给模型几个函数”，而是：

把模型意图转成真实动作之前的一层可观察、可拦截、可恢复的协议网关。

四、人类控制应该是 Agent 的一等能力

我们一路做了几个运行控制能力：

  • follow-up：Agent 忙时，新的输入进入队列。
  • steer：运行中给当前 run 注入引导。
  • stop：在安全点停止当前输出。
  • approve / deny：高影响工具执行前审批。

这些能力看起来是 UI 交互，但真正应该放在 Runtime 层。

原因是 TUI、Web、Telegram 都可能触发这些操作。如果把逻辑写在某个 UI 里，后面一定会重复和分叉。

比较合理的分层是：

控制状态放 core，比如 AgentRunControl；调度放 app/runtime，比如 AgentRunCoordinator；输入分流放 UI；状态展示走 event。

这里我最大的感受是：Agent 不是“等它回答完”。一个真正可用的 Agent 应该允许用户在中途介入、纠偏、排队、停止和审批。

五、扩展优于修改核心

做 Agent 系统时，很容易不断往 AgentLoop 里加 if-else：

  • 长期记忆要注入。
  • Skill 要注入。
  • 工具结果要卸载。
  • 审批要拦截工具。
  • steer 要临时插入消息。
  • Room 要注入共享消息。

如果这些都写进 AgentLoop，核心很快会变成一个巨大的业务汇总文件。

所以更好的方式是：

  • 用 Hook 做运行过程扩展。
  • 用 Extension 注册工具、Hook 和事件处理器。
  • 用 Event 让 UI 和日志观察过程。
  • 用 Runtime 装配具体能力。

这样 AgentLoop 只保留主循环：

长期记忆、系统提醒、工具卸载、审批、Skill、MCP、Todo、Schedule 都可以作为外部能力挂进来。

我的理解是：

AgentLoop 应该像发动机，不应该像后备箱。能通过扩展挂上的能力，就不要塞进核心循环。

六、UI 不是壳，它决定 Agent 是否可用

这个项目从 TUI 做到 Web，再做到 Telegram IM，我发现 UI 不是简单展示层。

TUI 需要：

  • 流式 token 展示。
  • Markdown 表格渲染。
  • 工具调用状态。
  • slash command。
  • token/context footer。

Web 需要：

  • 多 session 切换。
  • SSE 事件过滤。
  • 工具调用折叠。
  • 审批按钮。
  • Todo 便签。
  • Archive 页面。
  • Room 聊天室。
  • 模型切换下拉。

Telegram 需要：

  • 长文本截断。
  • 工具事件摘要。
  • 审批命令。
  • 每个 chat 绑定独立 runtime。

这些都不是“美化”。它们决定用户能不能理解 Agent 在干什么、卡在哪里、是否需要审批、上下文是否快满、当前用的模型是什么。

尤其是 Web 多 session 并行后，如果 SSE 事件不按 sessionId 过滤，A 会话的 token 就可能串到 B 页面。这个问题说明 UI 必须理解 Runtime 状态，而不是只做字符串输出。

七、后台任务和定时任务不是一回事

我们讨论过一个典型问题：用户说“一分钟后告诉我今天新闻”，模型容易调用 bash sleep 60。

这其实暴露了 Agent 对“时间”的误解。

我现在会把任务分成两类：

1. Background task

适合系统维护动作，比如：

  • 长期记忆抽取。
  • Todo 扫描。
  • 简单 reminder。

这类任务由 handler 执行，不一定需要 Agent 重新推理。

2. Schedule

适合“到点让 Agent 做一件事”，比如：

  • 每天 12 点总结新闻。
  • 每周五生成周报。
  • 明天提醒我检查某个 PR。

这类任务不应该写成后台 handler，而应该到点后自动提交一条 user 消息，重新进入普通 Agent 链路。

这个区别很关键：

Background 是系统自己做事；Schedule 是系统到点唤醒 Agent，让 Agent 像收到用户消息一样做事。

这样代码更少，语义也更清晰。

八、Plan 和 Team 是两种不同的多 Agent 思路

后来我们做了 /team 和 /plan，它们看起来都和多 Agent 有关，但定位不同。

Team：固定 DAG，用来探索

Team 更像“固定分工的小组”：

  • 多个 reader 并行读不同代码。
  • 多个 reviewer 并行看风险。
  • 最后把材料交回主 Agent。

它适合代码探索、架构理解、风险扫描。它不应该直接生成最终回答，而应该让主 Agent 根据所有材料整理输出。

所以它的链路是：

Plan：动态 DAG，用来执行

Plan 更像“针对目标生成任务图”：

  • 先生成 DAG。
  • 用户确认 /start。
  • 根据依赖并行执行可并行节点。
  • 对写文件、命令等高风险节点做串行或锁保护。
  • 最后让主 Agent 整理结果。

所以 Plan 的重点不是多角色，而是依赖关系和执行顺序。

我的理解是：

Team 解决“让不同视角并行探索”；Plan 解决“把目标拆成可执行依赖图”。

这两个能力可以复用 DAG runner，但产品语义不同，不能混为一谈。

九、多人聊天室让我重新理解多 Agent

真正的多 Agent 不只是“启动几个子 Agent 并行跑工具”。我们后来讨论的 Room 更像一个 LLM 聊天室：

  • 一个 room 有共享 hub messages。
  • 每个 Agent 有自己的 name、role、alias、system prompt 和工具白名单。
  • 用户可以 @某个Agent，也可以 @all。
  • Agent 的最终回复写入房间消息。
  • 后加入的 Agent 能看到当前房间共享消息。
  • 每个 Agent 又有自己的私有 session。

这里最重要的是“共享消息”和“私有上下文”分离。

如果所有 Agent 共用一个上下文，它们会互相污染；如果完全不共享，它们又不知道房间里发生了什么。

所以更合理的结构是：

@all 时多个 Agent 可以并行回复，但写回房间时不能按完成时间乱序，而是用成员顺序或 replyIndex 稳定写回。

这让我对多 Agent 的理解变成：

多 Agent 的核心不是“多个模型实例”，而是消息拓扑、角色边界、共享记忆和写回顺序。

十、Session 隔离是产品能力，不只是存储能力

Web 后来支持了多个 session 同时运行。用户切到 B 时，A 还可以继续跑；用户继续给 B 发消息，A/B 互不影响。

这要求：

  • 每个 session 有自己的 AgentRuntime。
  • 每个 runtime 有自己的 queue、control、approval、context cache。
  • 共享的东西只保留稳定配置，比如 provider、prompt、工具注册方式。
  • Web SSE 按 session 过滤。

这样一来，session 不只是 JSONL 文件名，而是一个完整运行态边界。

这也是为什么模型切换要放在 runtime，而不是全局 provider 上。A 可以用 flash，B 可以用 pro；切换模型只影响当前 session 后续请求，不应该让 snapshot 失效，也不应该改变摘要策略。

十一、我现在如何定义 Agent

做完这些之后，我会这样定义 Agent：

Agent 是一个以 LLM 为推理核心、以上下文窗口为状态输入、以工具协议为行动接口、以事件流为观察机制、以 session 为事实源、以 runtime 为调度边界的自动化系统。

它不是一个 prompt。

它也不是一个简单的工具调用框架。

它更像一套小型操作系统：

  • LLM 是 CPU。
  • Context 是内存窗口。
  • Session 是磁盘日志。
  • Tool 是外设和系统调用。
  • Hook 是内核扩展点。
  • Event 是日志和中断。
  • Runtime 是进程调度器。
  • UI 是人机接口。

这个类比不完全准确，但能帮助我判断架构边界：不要把所有东西塞进 LLM，也不要把所有东西塞进 AgentLoop。该落在 runtime 的落 runtime，该落在 context 的落 context，该落在 UI 的落 UI。

十二、我从这个项目里得到的几个原则

1. 核心链路要少

AgentLoop 只应该关心主循环和协议，不应该关心 Web、IM、Todo、Room、Memory 等业务。

2. 历史要完整，上下文要克制

Session 保存完整历史；Context 只构造本轮需要的窗口；动态提醒只临时注入。

3. 工具要可控

工具调用要可见、可审批、可拒绝、可恢复。高影响工具不能直接放行。

4. 事件要统一

TUI、Web、Telegram 不应该各自理解一套运行状态，而应该消费统一 AgentEvent。

5. 扩展优于修改核心

新增能力优先考虑 Hook、Extension、ToolRegistry、EventHandler、Runtime 装配，而不是改 AgentLoop。

6. 多 Agent 先想消息关系

多 Agent 的第一问题不是开几个线程，而是谁看到什么消息、谁保存什么上下文、谁按什么顺序写回。

7. 用户控制是安全边界

stop、steer、follow-up、approve、deny 都不是锦上添花，而是让 Agent 可用、可信、可控的基础。

结语

这一路从教学模式里的一个个问题开始：怎么 stop，怎么 steer，怎么排队，工具怎么注册，session 怎么删，Web 怎么展示，IM 怎么接入，定时任务怎么做，Plan 和 Team 有什么区别，多 Agent 聊天室怎么设计，模型怎么切换。

每个问题看起来都很小，但拼起来之后，我对 Agent 的理解越来越清晰：

Agent 的难点不只是让模型“想得对”，而是让系统在模型想错、工具失败、上下文变长、用户中途介入、多会话并行、多 Agent 同时回复时，仍然保持结构清晰、协议正确、状态可恢复。

所以我现在更愿意把 Agent 看成一个 Runtime 工程，而不是一个 Prompt 工程。

Prompt 很重要，但 Prompt 只是入口之一。真正让 Agent 可靠运转的，是上下文、工具、事件、会话、调度、审批和 UI 这些工程边界。

这也是 Aster 这个教学版项目最有价值的地方：它没有直接追求“大而全”的生产平台，而是把一个 Agent Runtime 应该具备的关键骨架拆开，让每个能力都能被看见、被理解、被修改。

从 Aster 看 Agent Harness：我对 Agent 的重新理解

发表于2026-06-08|更新于2026-06-09

我最早做 Aster 时，注意力放在模型上：prompt 怎么写，工具 schema 怎么描述，flash 和 pro 该怎么选。

项目从 TUI 做到 Web、Telegram、多 session、HITL、上下文压缩、Plan、Team、Room 之后，我开始看另一件事：模型外面的运行时有没有把边界兜住。

我把这组边界叫 Harness。它管上下文、工具协议、事件、审批、队列、停止、恢复、扩展、多入口和多 Agent 消息。LLM 负责生成。Harness 决定模型什么时候生成、能看什么、能动什么、用户怎么插手、系统出错后怎么收场。

一、我说的 Harness

现在我会把 Agent 拆成两层：

只接一个 chat model，也能做出 demo。用户连续用几轮之后，问题会立刻冒出来：

  • 当前 run 没结束，用户又发了一条消息怎么办。
  • 模型跑偏了，用户能不能中途纠正。
  • 用户按停止后，停在哪个点才不破坏 session。
  • bash/write/edit 要不要先让人确认。
  • 工具失败后，下一轮模型还能不能收到合法的 tool result。
  • 会话变长后，历史怎么压缩。
  • 服务重启后，上一次压缩进度还在不在。
  • Web 切到 B 会话时，A 会话要不要继续跑。
  • 多个 Agent 并行回复时，聊天室按什么顺序写消息。

这些问题靠 prompt 很难解决。代码里必须有对应的状态、协议和安全点。

二、人类控制要进入 Runtime

我最早做运行中控制时，拆了三个入口：follow-up、steer、stop。UI 可以放按钮和输入框，但 Runtime 必须保存控制状态。

follow-up：忙碌时排队

Agent 正在等待模型流式输出，或者正在跑工具。用户这时又发消息，直接拒绝会很难用。Aster 现在让普通输入走队列：

用户不用盯着 Agent 结束，也不用重新组织已经想到的问题。

steer：运行中纠偏

steer 不能当普通 user message 写进 session。当前 assistant 如果刚产生了 tool_calls，中间插入一条 user message，下一轮请求会破坏 assistant.tool_calls -> role=tool 的协议顺序。

Aster 把 steer 当成运行时控制信号：

这条消息只影响下一次模型请求，不污染 JSONL 历史。

stop：只在安全点停

stop 只设置标记。AgentLoop 在安全点检查标记，然后决定退出。

安全点包括：

  • LLM 请求前。
  • LLM stream chunk 后。
  • 工具调用前。
  • 工具结果写回后。
  • 下一轮 LLM 请求前。

如果系统在 assistant tool_call 后直接断掉，却没有写 role=tool，下一轮模型请求会看到坏协议。用户可以停 Agent，Harness 要保证停完还能恢复、继续和审计。

HITL：审批写进工具协议

bash/write/edit 属于高影响工具。模型调用这类工具时，UI 需要展示：

  • 工具名。
  • 参数。
  • call id。
  • 单个批准或拒绝。
  • 不带 id 时批量批准或拒绝。

拒绝也要写回合法 tool result。模型需要知道“用户拒绝了执行”。系统不能让模型一直等工具结果。

三、扩展先找挂载点

AgentLoop 很容易被写成大杂烩。

普通对话只需要：

加上工具以后变成：

再加长期记忆、Skill、审批、工具结果卸载、steer、system reminder、Room 共享消息，如果每个能力都塞进 AgentLoop，主循环会变得很难读，也很难测。

我给 Aster 定的边界是：

新增能力时，先看能不能挂到这些点上。

  • 长期记忆注入走 LLM 请求前 Hook。
  • steer 注入走 LLM 请求前 Hook。
  • 工具审批走工具调用前 Hook。
  • 工具结果卸载走工具结果写回前 Hook。
  • Room 共享上下文走 Room Agent 的请求前 Hook。
  • MCP、Skill、Todo、Schedule 走 Extension 注册工具。

MCP 名字冲突也在这个边界里处理。Aster 暴露给模型的 MCP 工具统一加 mcp_ 前缀，调用远端 MCP server 时仍使用原始工具名。这样本地工具命名空间不会撞上远端工具，MCP 协议也不用改。Web 侧只展示 MCP server 的 loaded/failed 状态，不把几十个工具名堆进侧栏。

Skill 也是同样的处理方式。完整 Skill 目录放到 workspace/skills。runtime 重启后扫描 name/description，模型需要时再用 load_skill 读取全文。核心循环不关心每个 Skill 的业务内容，只关心“这个能力能不能被加载、能不能被观察、能不能被审计”。

AgentLoop 只保留这条主线：

核心循环稳住后，后面的能力才不会互相拖拽。

四、Session 恢复和上下文压缩

长会话很快会把 naive 实现打穿。

每轮请求都从 JSONL 全量加载，再把历史塞给模型，会带来几个问题：

  • token 成本上升。
  • 请求变慢。
  • 上下文超过模型限制。
  • 工具调用多轮 replay，浪费明显。

我现在把 session 和 context 分开看。

Session 是事实源

Session JSONL 保存完整原始历史：

  • user message
  • assistant message
  • tool calls
  • tool results
  • run started / finished
  • usage / event

它要完整、可审计。摘要、长期记忆和当前时间不应该直接写进这里。

Context 是请求窗口

Context 只表示本轮请求给模型看的窗口。Aster 用 ContextWindowCache 维护：

更早的对话压成摘要，最近几轮完整保留，当前 user 输入完整保留。模型还能看到旧对话的结论，也不会被完整历史撑爆。

system-reminder 是临时注入层

长期记忆、旧对话摘要、当前时间、Skill 提示都属于动态内容。Aster 把它们放到最后一条 user 消息开头：

它只影响本轮请求，不落入永久 session。

Snapshot 保存压缩进度

内存 cache 解决不了重启。服务恢复后，如果再次从 JSONL 全量 replay，就会重复摘要已经处理过的历史。

ContextWindowSnapshot 保存上一轮的上下文窗口进度：

恢复时走这个流程：

这里有个细节：切换 chat model 不应该让 snapshot 失效。deepseek-v4-flash 和 deepseek-v4-pro 是当前 session runtime 的运行态选择。摘要策略没变，prompt 没变，seq/hash 没变，恢复断点就应该继续有效。

五、工具调用是一条协议链

模型返回 tool_calls，看起来像函数调用。项目里要处理的是完整链路：

每个环节都要保证下一轮模型能继续。

  • 工具成功，写 role=tool。
  • 工具失败，也写 role=tool。
  • 审批拒绝，也写 role=tool。
  • 大结果卸载，保留 recordId、path、preview。
  • 多个工具并行执行，写回时按 call id 配对。

工具系统在项目里表现为一套由 Harness 管理的能力协议。这也是我一直要求扩展工具走当前项目注册方式的原因。

六、Team 和 Plan 是两条路

/team 和 /plan 都会启动多个 Agent 或多个任务，但它们解决的问题不同。

Team：固定 DAG，用来探索

Team 像一个固定小组。比如 3 个 reader 读不同模块，2 个 reviewer 看风险，最后把材料交回主 Agent。

它适合：

  • 理解代码结构。
  • 扫描架构风险。
  • 分头读不同文件。
  • 汇总 reviewer 视角。

Aster 里 Team 的链路是：

Team 不直接替主 Agent 下最终结论。它负责把现场材料读出来。

Plan：动态 DAG，用来执行

Plan 面向任务拆解和执行控制：

Plan 要关心：

  • 哪些节点能并行。
  • 哪些节点必须等前置任务。
  • 哪些节点只读文件。
  • 哪些节点会写文件或执行命令。
  • 写操作要不要串行或加锁。
  • 高影响工具是否继续走 HITL。

Team 和 Plan 可以复用 DAG runner，但产品语义要分开。Team 解决探索，Plan 解决执行。

七、多人聊天室的重点是消息拓扑

我以前想多 Agent，容易想到“多开几个子 Agent”。Room 聊天室做完后，我更关心消息拓扑。

这样后加入的 Agent 可以看到房间历史，每个 Agent 又有自己的私有上下文。它们共享房间消息，不共享各自的内部历史。

@all 按触发顺序写回

用户发 @all 时，多个 Agent 可以并行跑。谁先完成不重要，写回顺序要稳定。

Aster 在触发时记录成员顺序和 replyIndex：

这和并行工具调用很像。工具可以并发跑，tool result 通过 tool_call_id 对回去。Room Agent 可以并发回复，房间消息通过 replyIndex 排回去。

移除 Agent 归档成员关系

从聊天室移除某个 Agent，只归档这个房间里的成员关系：

全局 Agent 配置还在。恢复时 generation 增加，后续使用新的私有 session，旧上下文不继续混用。

这套设计把三件事分清楚了：角色配置、房间共享消息、Agent 私有上下文。

八、多入口和多 session 需要运行态隔离

Aster 有 TUI、Web、Telegram 三个入口。它们都进入同一套 Runtime：

/stop、审批、Plan、Team、schedule、background 通知都不用在三个入口重复实现。

Web 又加了一层 session runtime pool：

用户切到 B 时，A 可以继续跑。用户继续给 B 发消息，A/B 互不影响。

这个能力要求 Harness 隔离这些状态：

  • queue
  • stop/steer control
  • pending approvals
  • context cache
  • 当前 chat model
  • SSE 事件过滤

这里还有个 UI 空态细节。Web 启动时不要自动创建 default session，也不要自动创建默认聊天室。没有会话就展示引导。用户点击新建，或者第一次发送消息时，再同步创建 session。这样 workspace 里的数据都来自用户行为，不会混进页面占位数据。

九、Background 和 Schedule 的边界

用户说“一分钟后告诉我今天新闻”，模型曾经选择 bash sleep 60。这暴露了一个问题：模型不知道系统里哪个任务能力适合延时，哪个适合到点自动执行。

后来我把它拆成两类。

Background task

Background 适合系统维护动作：

  • 长期记忆抽取。
  • Todo 扫描。
  • 简单 reminder。

handler 自己能完成的事，没必要再唤醒 Agent 推理。

Schedule

Schedule 适合到点让 Agent 做事：

  • 每天 12 点总结新闻。
  • 明天提醒我检查任务。
  • 周五生成周报。

Schedule 到点后自动提交一条 user message，重新走普通 Agent 链路：

Web 做定时任务面板时也沿用这套状态源。页面表单、Agent 的 schedule 工具、调度器，都读写同一个 ScheduledUserMessageManager 和 schedules.json。

十、模型切换也是运行态

模型名放错位置，会影响后面的恢复和多 Agent。

现在 Aster 按这几条处理：

  • Provider 表示 API 服务配置。
  • 当前 chat model 属于 runtime/session 状态。
  • AgentLoop 每次 LLM 请求前读取当前模型。
  • 已经发出的 SSE 请求不打断。
  • 下一次 LLM 请求使用新模型。
  • 摘要策略不跟随 chat model 变化。
  • snapshot 不因为 chat model 变化失效。

多 Agent 里再细分：

  • Team 一次性探索运行，不指定模型时跟随当前 chat model；指定 /team --model ... 时，本次运行启动时快照模型，所有成员一致。
  • Room Agent 是长期配置对象，模型属于 RoomAgentProfile，不同 Agent 可以用不同模型。
  • Plan-and-execute 分成两段，planner 固定 pro，worker 固定 flash。
  • 主 Agent 最终整理使用当前 session runtime 的 chat model。

模型名不能当全局常量处理。它和 session、room、team、plan 的运行状态绑定。

十一、LLM 层按能力拆

做多模态时，我重新整理了 llm 层。项目现在同时出现这些能力：

  • 文本对话：DeepSeek / OpenAI-compatible SSE。
  • 本地文本模型：Ollama qwen3:latest。
  • Embedding：Ollama nomic-embed-text:v1.5。
  • 图片理解：Ollama llava-llama3:latest。
  • 后续可能接语音转文字、文字转语音、图片生成、多模态模型。

单一 chat client 容易把这些协议混在一起。Aster 现在按能力拆包：

能力接口放在能力包根目录：

request / response / value object 放到各自的 model/ 子包：

Embedding、语音、图片、多模态的协议形状不同。先按能力拆清楚，后面接供应商时才知道一个类属于哪条链路。

多模态先走独立分支

Web Chat 现在可以上传图片，走 Ollama 多模态模型流式回答：

这条链路目前不写普通 session，不触发工具，不参与上下文压缩。

如果图片直接进入 AgentLoop，需要同时改很多协议：

  • Message.content 要支持结构化内容。
  • session JSONL 要决定保存图片引用还是 base64。
  • 上下文压缩要知道怎么处理图片。
  • 工具调用前后要处理多模态 content。
  • <system-reminder> 要重新确认注入位置。
  • RAG、Room、Team、Plan 要不要同步支持图片。

第一版先把 llm/multimodal 能力层跑通，再用 Web 独立分支验证模型、流式和错误边界。等图片进入 AgentLoop，再补 session、context、tool protocol 和压缩策略。

RAG 也采用类似思路。Knowledge 页面第一版独立链路：

它有自己的 session 和来源展示，但不开放普通 Agent 工具。

本地模型也要约束

llava-llama3 有个小问题：用户问“中文回答”，模型仍可能用英文描述图片。

多模态服务里加了 system 语言约束：

当用户问题包含中文时，user content 里再补一句“请严格使用简体中文回答”。本地模型接上以后也要受 Harness 约束，不能只靠模型自觉。

十二、我现在怎么拆 Agent Harness

我现在看 Aster，会按这几组拆：

这个拆法比一句“Agent Runtime”更有用。排查问题时，我能先判断问题落在哪个 Harness：

  • 模型没看到记忆，查 Context Harness。
  • 工具审批没回来，查 Tool Harness 和 Control Harness。
  • Web 切会话串消息，查 Session Harness 和 UI Harness。
  • Room @all 顺序乱，查 Multi-Agent Harness。
  • 图片回答不按中文，查 LLM Harness。

十三、我从 Aster 得到的几条规则

1. 人类控制要进 Runtime

stop、steer、follow-up、approve、deny 都是运行态能力。UI 负责触发，Runtime 负责状态和安全点。

2. 新能力先找扩展点

优先考虑 Hook、Extension、ToolRegistry、EventHandler、Runtime 装配。直接改 AgentLoop 前，先说明为什么现有挂载点不够。

3. Session 和 Context 分开

Session 保存完整历史。Context 构造本轮请求窗口。snapshot 恢复压缩进度。system-reminder 临时注入动态信息。

4. 工具协议必须闭环

工具成功、失败、拒绝、卸载，都要写回合法 tool result。下一轮模型请求不能看到半截协议。

5. Team 和 Plan 分开设计

Team 是固定 DAG 探索，多视角并行读代码。Plan 是动态 DAG 执行，核心是依赖解析和任务调度。

6. 多 Agent 先画消息拓扑

先决定谁看什么消息、谁保存什么上下文、谁按什么顺序写回，再决定开几个 Agent 并行。

7. 多 session 要有独立 runtime

每个 session 有自己的 queue、control、approval、context cache 和当前模型。切换 UI 不应该打断旧 session。

8. 模型属于运行态

模型切换只影响后续请求，不破坏 session、摘要策略和上下文恢复。

9. LLM 层先按能力拆

Text、Embedding、Speech、Image、Multimodal 的协议形状不同。先拆能力接口和 model 子包，再决定哪些能力进入 AgentLoop。

10. 独立 LLM 能力也要保持 Harness 标准

RAG 和图片理解可以先不走 AgentLoop，但仍要有 session、来源、错误边界和流式输出。

结语

做完这些功能后，我不再把 Agent 当成 prompt 工程。

Prompt 仍然重要，但很多时间花在模型输出之外：用户打断、工具审批、会话恢复、上下文压缩、多 session 并行、多 Agent 写回顺序、计划任务依赖、UI 状态解释。

Aster 的价值在这里。它没有把这些能力藏进一个大类，而是把 Harness 的骨架拆出来。你能看到每个能力怎么进 Runtime，怎么过协议，怎么被 UI 观察，怎么在重启后继续。

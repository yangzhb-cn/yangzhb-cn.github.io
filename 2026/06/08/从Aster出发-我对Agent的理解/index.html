从Aster出发：我对Agent的理解

发表于2026-06-08|更新于2026-06-08

第一次把 Aster 从 TUI 往 Web、Telegram、多会话、工具审批、上下文压缩和计划执行推进时，我才发现，自己之前把 Agent 看窄了。

我现在会先看 Runtime，再看模型。模型负责推理、生成和选择工具；Runtime 负责把一次推理接进真实系统：上下文从哪里来，工具怎么执行，用户怎么审批，失败怎么写回，会话怎么恢复，UI 怎么知道当前发生了什么。

Aster 这个教学版项目给我的价值在这里。它先把 Agent Runtime 的骨架拆开：LLM、Context、Tool、Session、Event、Runtime、UI。你能看到每一层怎么工作，也能看到边界一乱，系统会在哪里出问题。

一、我先把 Agent 看成一条循环

最小的 AgentLoop 大概长这样：

最容易踩坑的是消息协议。

模型返回 assistant.tool_calls 后，下一步必须写入对应的 role=tool 结果，tool_call_id 也要对上。工具失败、用户拒绝审批、工具结果太大需要卸载，这些情况都要写成合法的 tool result。你不能随手插一条普通用户消息进去糊弄下一轮请求。

Aster 让我意识到一件很具体的事：LLM 可以答错，Runtime 不能把协议写乱。协议一乱，下一轮请求可能直接 400；更麻烦的情况是请求没报错，但模型读到一段歪掉的历史，然后开始沿着错误上下文继续推。

二、我把上下文拆成三层

一开始写 Agent，很多人会把上下文处理成“把历史消息拼进 prompt”。会话短的时候这招能跑。会话一长，问题立刻冒出来：

  • 每轮都从 JSONL 全量 replay，浪费时间。
  • 历史撑爆模型上下文。
  • 摘要如果写回 session，会污染审计历史。
  • 动态信息塞进 system prompt，会和静态规则混在一起。

我后来把上下文拆成三层：

  1. 完整历史：JSONL session，记录事实和审计信息。
  2. 运行窗口：ContextWindowCache，保存旧对话摘要和最近完整 turn。
  3. 临时注入：<system-reminder>，只在本轮请求里注入长期记忆、旧对话摘要、当前时间和 Skill。

这三个东西要分清楚。

Session 回答“发生过什么”。Context 回答“这一轮模型该看什么”。System reminder 回答“这一轮还要提醒模型什么”。你把它们揉在一起，后面想恢复、审计、压缩都会变麻烦。

Aster 后面又加了 ContextWindowSnapshot。重启后，Runtime 不用重新摘要整份 JSONL；它先校验 lastSeq/lastHash，对得上就恢复 runningSummary + recentTurns，再用 loadMessageRecordsAfter(lastSeq) 补齐新增消息。

我现在判断长会话 Agent 时，会先看它有没有把持久历史和请求窗口分开。历史要完整，窗口要克制。这两个目标混在一个结构里，迟早会互相拖累。

三、工具调用需要协议和审批

工具调用表面上像函数调用。放进 Agent 系统之后，它会牵出一整套协议：

  • 工具 schema 怎么暴露给模型。
  • 参数怎么解析和校验。
  • 工具能不能并行执行。
  • 结果太大时怎么卸载。
  • 高风险工具怎么走人工审批。
  • 失败后怎么写回合法 tool result。
  • UI 怎么展示名称、参数、耗时和结果。

bash/write/edit 这类工具尤其不能直接放行。模型一旦拿到它们，Runtime 就必须给用户拦截点。HITL 审批不该只写在 UI 里，它应该是 Runtime 的安全边界。

审批时，用户要看到工具名称、参数和 call id。用户可以批准一个、拒绝一个，也可以批量处理。无论用户批准还是拒绝，Runtime 都要给模型写回结果。模型需要知道这次工具到底执行了没有，然后继续完成回答。

我现在会把工具层看成一个网关：模型提出动作，Runtime 把动作变成可观察、可拦截、可恢复的执行记录。

四、用户要能在运行中接管

Aster 做过几类运行控制：

  • follow-up：Agent 忙的时候，新输入先进队列。
  • steer：运行中给当前 run 注入引导。
  • stop：在安全点停止当前输出。
  • approve / deny：高影响工具执行前让用户决定。

这些能力看起来像交互细节，放错层就会出问题。TUI、Web、Telegram 都可能触发这些操作。如果你把逻辑写在某个 UI 里，后面每接一个入口就要复制一份。

我更愿意这样分层：

AgentRunControl 保存控制状态，AgentRunCoordinator 负责调度，UI 只做输入分流和状态展示，事件走 AgentEvent。

一个能长期用的 Agent，不能要求用户坐等它输出完。用户要能中途纠偏、排队、停止、审批。Runtime 把这些动作变成一等能力，UI 才不会各写各的控制逻辑。

五、核心循环要瘦

做 Agent 系统时，你很容易往 AgentLoop 里塞逻辑：

  • 注入长期记忆。
  • 注入 Skill。
  • 卸载大工具结果。
  • 拦截高风险工具。
  • 插入 steer 消息。
  • 给 Room 注入共享消息。

这些 if-else 写进去之后，AgentLoop 很快会变成一个大杂烩。

Aster 后面用 Hook、Extension、ToolRegistry、EventHandler 和 Runtime 装配来拆这些能力。AgentLoop 只保留主链路：

长期记忆、系统提醒、工具卸载、审批、Skill、MCP、Todo、Schedule 都能挂在外面。核心循环越瘦，你越容易判断问题发生在哪一层。

我会把 AgentLoop 当发动机看。发动机只负责转，其他能力该挂在外部就挂在外部。

六、UI 直接影响可用性

Aster 从 TUI 做到 Web，再接 Telegram，我对 UI 的看法也变了。UI 要把 Runtime 状态翻译给用户。

TUI 要处理流式 token、Markdown 表格、工具调用状态、slash command 和 token/context footer。

Web 要处理多 session 切换、SSE 事件过滤、工具调用折叠、审批按钮、Todo 便签、Archive 页面、Room 聊天室和模型切换。

Telegram 要处理长文本截断、工具事件摘要、审批命令，以及每个 chat 对应的独立 runtime。

这些功能直接决定用户能不能看懂 Agent 正在做什么。它卡在哪里？是不是在等审批？上下文是不是快满了？当前用的是哪个模型？这些信息不展示出来，用户只能猜。

Web 多 session 并行时，SSE 事件过滤尤其关键。用户切到 B 会话后，A 会话还可能继续跑。如果 UI 没按 sessionId 过滤，A 的 token 会串到 B 页面。这个 bug 说明 UI 必须理解 Runtime 状态，不能只做输出面板。

七、后台任务和定时任务分开处理

我们讨论过一个典型场景：用户说“一分钟后告诉我今天新闻”，模型可能会调用 bash sleep 60。

这说明模型把“等待”和“调度”混在了一起。Runtime 应该接管这件事。

我现在把任务分成两类。

1. Background task

这类任务适合系统维护动作：

  • 长期记忆抽取。
  • Todo 扫描。
  • 简单 reminder。

handler 可以直接执行它们，不需要 Agent 重新推理。

2. Schedule

这类任务适合到点后让 Agent 重新工作：

  • 每天 12 点总结新闻。
  • 每周五生成周报。
  • 明天提醒我检查某个 PR。

Runtime 到点后提交一条新的 user 消息，让它进入普通 Agent 链路。代码少，语义也清楚：系统维护动作走 Background，需要模型判断的事走 Schedule。

八、Team 和 Plan 的分工

Aster 后面做了 /team 和 /plan。它们都和多 Agent 有关，但解决的问题不同。

Team：固定分工，用来探索

Team 更像一个临时小组：

  • 多个 reader 并行读不同代码。
  • 多个 reviewer 并行看风险。
  • 主 Agent 收集材料后整理回答。

它适合代码探索、架构理解和风险扫描。Team 的输出不该直接变成最终回答，主 Agent 需要拿回材料，再按用户问题组织结果。

Plan：生成任务图，用来执行

Plan 先把目标拆成 DAG，再等用户确认 /start。执行时，Runtime 根据依赖并行跑可并行节点；写文件、跑命令这类高风险节点要串行，或者加锁保护。最后主 Agent 再整理结果。

Team 解决视角并行，Plan 解决依赖执行。它们可以复用 DAG runner，但产品语义不能混在一起。

九、Room 让我重新看多 Agent

多 Agent 要先从消息关系看起。Aster 里的 Room 更像一个 LLM 聊天室：

  • 一个 room 有共享 hub messages。
  • 每个 Agent 有自己的 name、role、alias、system prompt 和工具白名单。
  • 用户可以 @某个Agent，也可以 @all。
  • Agent 的最终回复写入房间消息。
  • 后加入的 Agent 能看到当前房间共享消息。
  • 每个 Agent 保留自己的私有 session。

共享消息和私有上下文要分开。所有 Agent 共用一个上下文，它们会互相污染；完全不共享，它们又不知道房间里刚发生了什么。

我更认可这个结构：

@all 时多个 Agent 可以并行回复。写回房间时，Runtime 不能按完成时间乱排，而要按成员顺序或 replyIndex 稳定写回。否则用户每次看见的对话顺序都不一样，后面的 Agent 也会读到不稳定的上下文。

多 Agent 的第一问题是消息关系：谁能看见什么，谁保存什么，谁按什么顺序写回。

十、Session 是运行边界

Web 支持多个 session 同时运行后，session 已经超过一个 JSONL 文件名。

用户切到 B 会话时，A 会话可以继续跑。用户继续给 B 发消息，A/B 不能互相影响。要做到这一点，每个 session 都需要自己的 AgentRuntime，也需要自己的 queue、control、approval、context cache。

共享部分只保留稳定配置，比如 provider、prompt 和工具注册方式。Web SSE 必须按 session 过滤。

模型切换也应该放在 runtime 上，别挂到全局 provider 上。A 可以用 flash，B 可以用 pro。切换模型只影响当前 session 后续请求，不应该让 snapshot 失效，也不应该改变摘要策略。

我现在会把 session 看成运行态边界。只要它还在跑，就有自己的队列、控制器、审批状态和上下文窗口。

十一、我现在怎么定义 Agent

一句定义：

Agent 是一套围绕 LLM 运行的自动化系统。LLM 负责推理，Context 提供状态输入，Tool 提供行动接口，Event 暴露运行过程，Session 保存事实，Runtime 负责调度，UI 让用户看见并接管。

操作系统这个类比对我有帮助：

  • LLM 像 CPU。
  • Context 像内存窗口。
  • Session 像磁盘日志。
  • Tool 像外设和系统调用。
  • Hook 像内核扩展点。
  • Event 像日志和中断。
  • Runtime 像进程调度器。
  • UI 像人机接口。

类比不需要百分百准确。它提醒我别把所有东西塞进 prompt，也别把所有东西塞进 AgentLoop。该落在 runtime 的落 runtime，该落在 context 的落 context，该落在 UI 的落 UI。

十二、几个判断

1. 核心链路要少

AgentLoop 只处理主循环和协议。Web、IM、Todo、Room、Memory 这些能力放到外部扩展。

2. 历史要完整，上下文要克制

Session 保存完整历史。Context 只构造本轮窗口。动态提醒只做临时注入。

3. 工具要可控

工具调用要可见、可审批、可拒绝、可恢复。高影响工具不能直接放行。

4. 事件要统一

TUI、Web、Telegram 消费同一套 AgentEvent。每个 UI 不该各自发明运行状态。

5. 扩展优先

新增能力先考虑 Hook、Extension、ToolRegistry、EventHandler 和 Runtime 装配。改 AgentLoop 应该是最后选择。

6. 多 Agent 先看消息关系

先问谁看到什么消息、谁保存什么上下文、谁按什么顺序写回，再问开几个 Agent。

7. 用户控制要进 Runtime

stop、steer、follow-up、approve、deny 都要进 Runtime。它们决定用户能不能接管 Agent。

结语

Aster 的很多问题都很小：怎么 stop，怎么 steer，怎么排队，工具怎么注册，session 怎么删，Web 怎么展示，IM 怎么接入，定时任务怎么做，Plan 和 Team 怎么分，Room 怎么设计，模型怎么切换。

这些小问题拼在一起后，我对 Agent 的看法变得更工程化。模型想错、工具失败、上下文变长、用户中途介入、多会话并行、多 Agent 同时回复，这些情况都会发生。Runtime 要让系统在这些情况下还能保持协议正确、状态可恢复、用户能看懂。

Prompt 是入口。Aster 让我更关心入口后面的东西：上下文、工具、事件、会话、调度、审批和 UI。这些边界搭稳之后，Agent 才像一个能长期运行的系统。
